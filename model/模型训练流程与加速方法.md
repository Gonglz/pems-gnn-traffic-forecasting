本文件记录当前真实主线，而不是早期实验脚本。当前项目主线是：

- `model/train_sampler.py`
- `model/evidence_pack.py`
- `model/evidence_eval.py`

## 1. 当前训练任务

数据来自 graph-aligned PeMS arrays：

- `X_ext.npy`: `(T, N, 6)`
- `Y.npy`: `(T, N)`

- `step52_edge_index.pt`: PyG `edge_index`
- `step62_neighbors.pkl`: graph node metadata and neighbor payload

特征顺序：

2. `occupancy_interp`
3. `speed_interp`
4. `tavg`
5. `pcpn`
6. `is_weekend`

目标：`flow_interp`

预测 horizon：

- `t+5` minutes
- `t+15` minutes
- `t+30` minutes

## 2. 当前模型和训练入口

模型是 `MultiHeadRFGraphSAGEDyn`：

- 三个 head：`5min`, `15min`, `30min`
- 每个 head 使用 PyG `SAGEConv`
- 训练时一次 forward 生成三个 head 的输出，再合并 loss backward

典型单卡训练：

```bash
CUDA_VISIBLE_DEVICES=3 \
/scratch/lgong1/envs/traffic-env/bin/python -m model.train_sampler \
  --data-dir /scratch2/lgong1/finalproject_gpu_parallel_test/<run>/pems_data \
  --edge-index-path /scratch2/lgong1/finalproject_gpu_parallel_test/<run>/pems_data/step52_edge_index.pt \
  --checkpoint-dir /scratch2/lgong1/finalproject_gpu_parallel_test/<run>/checkpoints/single_gpu \
  --epochs 3 \
  --batch-size 1024 \
  --num-workers 4
```

典型双卡 DDP：

```bash
CUDA_VISIBLE_DEVICES=3,4 \
/scratch/lgong1/envs/traffic-env/bin/torchrun --standalone --nproc_per_node=2 \
  -m model.train_sampler \
  --data-dir /scratch2/lgong1/finalproject_gpu_parallel_test/<run>/pems_data \
  --edge-index-path /scratch2/lgong1/finalproject_gpu_parallel_test/<run>/pems_data/step52_edge_index.pt \
  --checkpoint-dir /scratch2/lgong1/finalproject_gpu_parallel_test/<run>/checkpoints/ddp_2gpu \
  --epochs 3 \
  --batch-size 1024 \
  --num-workers 4 \
  --ddp-shard-mode flat-batch
```

## 3. 当前有效加速手段

### 3.1 PyG NeighborSampler

训练不在每个 step 全图 forward，而是用 `NeighborSampler` 对 seed nodes 做邻居采样。

当前每个 horizon 的采样 fanout：

```python
SIZES = {
    "5min": [8, 8],
    "15min": [12, 12],
    "30min": [16, 16],
}
```

这比全图 dense 训练更适合 4,883 station 的中大规模实验，也和 PyG 主线保持一致。

### 3.2 AMP 混合精度

`train_sampler.py` 默认启用 CUDA AMP：

```python
from torch.cuda.amp import GradScaler, autocast

scaler = GradScaler(enabled=amp_enabled)
with autocast(enabled=amp_enabled):
    preds = forward_model(head_batches=head_batches)
    raw_losses = [...]
scaler.scale(loss).backward()
scaler.step(optimizer)
scaler.update()
```

可用 `--no-amp` 关闭。当前 Evidence Pack 默认使用 AMP。

### 3.3 TF32

CUDA 设备上启用 TF32 matmul：

```python
torch.backends.cuda.matmul.allow_tf32 = True
```

这对 V100 之后的 GPU 更有意义；在当前 V100S 上主要作为兼容性配置保留。

### 3.4 DDP 多卡训练

当前 DDP 使用 PyTorch `DistributedDataParallel`：

```python
model = DistributedDataParallel(model, device_ids=[local_rank])
```

为避免多 head forward 的 DDP 同步问题，当前实现不是三个 head 分别 backward，而是一次 forward 接收 `head_batches`，一次性返回三个 head 的预测，再合并 loss。

### 3.5 Fair flat-batch sharding

旧版 DDP 同时切 time 和 node，会导致 1 GPU / 2 GPU workload 不完全等价。P1 后默认使用：

```bash
--ddp-shard-mode flat-batch
```

含义是先构造全局 work items：

```text
(time_idx, seed_node_batch_idx)
```

然后按 rank stride 分片：

```python
rank_work = all_work[rank::world_size]
```

这样 1 GPU / 2 GPU 覆盖同一份 `(time, node batch)` 工作，只是被不同 rank 分走。

保留旧模式：

```bash
--ddp-shard-mode legacy-time-node
```

仅用于复现实验，不推荐作为新的 scaling 口径。

### 3.6 DataLoader workers and pinned memory

`NeighborSampler` 使用：

```python
num_workers=args.num_workers
pin_memory=torch.cuda.is_available()
```

当前 medium/fullproof 使用 `--num-workers 4`。

### 3.7 非 TTY 日志优化

full-scale 运行时，tqdm 的回车进度条会在 pipe 中产生大量无换行输出；如果外层 logger 按行读取，rank0 可能被 stdout pipe 卡住。当前修复为：

```python
if rank == 0 and (sys.stdout.isatty() or sys.stderr.isatty()):
    iterator = tqdm(...)
```

Evidence Pack 的 `run_logged` 也会每行 `flush()`，保证日志及时落盘。

## 4. Evidence Pack 编排

推荐通过 Evidence Pack 统一跑实验：

```bash
/scratch/lgong1/envs/traffic-env/bin/python -m model.evidence_pack run \
  --source-data-dir /scratch/lgong1/finalproject/pems_data \
  --output-root /scratch2/lgong1/finalproject_gpu_parallel_test/<run_id> \
  --epochs 3 \
  --batch-size 1024 \
  --num-workers 4 \
  --max-train-times 512 \
  --max-val-times 128 \
  --skip-rf \
  --ddp-shard-mode flat-batch \
  --single-gpu 3 \
  --ddp-gpus 3,4
```

DDP-only full-scale proof：

```bash
/scratch/lgong1/envs/traffic-env/bin/python -m model.evidence_pack run \
  --source-data-dir /scratch/lgong1/finalproject/pems_data \
  --output-root /scratch2/lgong1/finalproject_gpu_parallel_test/fullproof-<timestamp> \
  --all-times \
  --epochs 1 \
  --batch-size 1024 \
  --num-workers 4 \
  --max-train-times 0 \
  --max-val-times 0 \
  --max-eval-times 512 \
  --skip-rf \
  --skip-single \
  --ddp-shard-mode flat-batch \
  --ddp-gpus 3,4
```

输出结构：

```text

plots/

logs/
checkpoints/

```

其中 `predictions/prediction_samples_station_<id>.csv` 是 station/timestamp 级推理 demo。

## 5. Ablation 支持

### 5.1 without_weather

```bash
--feature-ablation without_weather
```

行为：

- 只替换 weather columns：`tavg`, `pcpn`
- replacement 只从 train split 计算
- train/eval 共用同一 metadata，避免配置错位

### 5.2 without_graph_neighbors

```bash
--graph-mode self_loop
```

行为：

- 用 self-loop-only `edge_index`
- 保留节点自身特征
- 移除跨 station graph neighbor 信息

## 7. custom CUDA / SpMM 的当前地位

项目里仍有 custom CUDA / GPU 加速实验：

- `model/spmm_ext.py`
- `model/spmm_kernel.cu`
- `data_process/step40_fill_gpu.py`
- 早期 Mahalanobis / cudf / numba 相关处理脚本

P1 前已经做过轻量修复：

- `spmm_ext.py` 支持 lazy JIT 和 fallback
- `spmm_kernel.cu` 使用 scatter-add atomic 修正聚合语义
- `dyn_conv_final.py` 修正 valid neighbor mean
- `step40_fill_gpu.py` 修正部分 GPU fill 兼容性问题

但这些不是当前 Evidence Pack 主线。原因：

- 当前训练主线已经使用 PyG `SAGEConv + NeighborSampler`
- custom SpMM 没有进入 P0/P1 benchmark
- 继续深挖 custom kernel 的边际收益低于模型质量、normalization、baseline 和报告证据

推荐叙事：

- 本项目讲真实数据 ML pipeline、PyG GNN、DDP、Evidence Pack、full-scale proof。
- 另一个 CUDA/C/C++ 项目讲底层 kernel 和性能优化。
